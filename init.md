เอกสารรวบรวมข้อมูลจำเพาะ (Knowledge Base / RAG) สำหรับพัฒนาระบบและ API Drug Allergy Card
ข้อมูลชุดนี้สรุปจากรายละเอียดการประชุม เพื่อใช้เป็นแหล่งข้อมูลอ้างอิงสำหรับทีมพัฒนา (Developer) ในการออกแบบสถาปัตยกรรมระบบ, ฐานข้อมูล, และ API Service ครับ
1. ตรรกะหลักของอัลกอริทึม (Core Logic & Data Pipeline)
เงื่อนไขการค้นหา (Trigger): ระบบจะทำงานเมื่อพบผู้ป่วยที่ต้องเข้านอนโรงพยาบาล (Admit) ด้วยอาการแพ้ยารุนแรง (เช่น สตีเวนส์-จอห์นสัน ซินโดรม)
การสืบค้นประวัติย้อนหลัง (Retrospective Search): ระบบจะต้องดึงข้อมูลประวัติการสั่งจ่ายยาย้อนหลัง 1 เดือน ก่อนวันที่เข้านอนโรงพยาบาล
การค้นหาข้ามเครือข่าย (Cross-Hospital Tracking): เป็น Requirement สำคัญที่ระบบต้องค้นหาประวัติการรับยาจาก ทุกโรงพยาบาล ไม่ใช่แค่โรงพยาบาลที่ผู้ป่วยแอดมิท เนื่องจากผู้ป่วยมักได้รับยาที่ทำให้แพ้จากโรงพยาบาลขนาดเล็ก หรือคลินิก ก่อนที่จะถูกส่งตัวมารักษาอาการแพ้ยาอย่างรุนแรงที่โรงพยาบาลขนาดใหญ่
การจัดกลุ่มข้อมูลยา (Drug Classification): ยาที่ค้นพบจะต้องถูกผูกเข้ากับรหัสมาตรฐาน TMT Code และเลขมาตรฐาน 24 หลัก
การจัดการกลุ่มยาที่คล้ายคลึง (Cross-Reaction): อัลกอริทึมต้องสามารถประมวลผลการแพ้แบบกลุ่ม (Similarity/Cross-reaction) เพราะหากแพ้ยา 1 ตัว อาจจะต้องบล็อกยาอื่นที่มีโครงสร้างคล้ายกันอีก 10-20 ตัว
ความถี่ในการรันระบบ (Batch Processing): ข้อมูลผู้ป่วยแพ้ยาจะเพิ่มขึ้นทุกวัน ระบบควรมีการรันสคริปต์เพื่ออัปเดตข้อมูลผู้ป่วยรายใหม่ (เช่น รันสัปดาห์ละ 1 ครั้ง)
2. ระบบการยืนยันข้อมูลโดยผู้เชี่ยวชาญ (Verification System)
ผู้ใช้งานหลัก (Users): แพทย์ และ เภสัชกรประจำโรงพยาบาล
การยืนยันตัวตน (Authentication): บังคับให้ผู้ใช้งานต้องล็อกอินเข้าสู่ระบบผ่าน Provider ID เพื่อให้สามารถตรวจสอบย้อนหลัง (Audit trail) ได้ว่าบุคลากรท่านใดเป็นผู้ยืนยันข้อมูล
User Interface (UI): อาจไม่ต้องพัฒนาหน้าจอใหม่ทั้งหมด แต่ใช้วิธีนำระบบนี้ไปปลั๊กอิน (Integrate) เข้ากับหน้าจอของระบบเดิมที่ใช้อยู่แล้วในโรงพยาบาล (เช่น "ระบบของบริษัทต่อ") เพื่อให้เภสัชกรใช้งานได้ทันที
การจัดการสถานะข้อมูล (Data State Management):
กรณีผู้ป่วยได้รับยาพร้อมกันหลายตัว (เช่น 5 ตัว) เภสัชกรจะต้องเป็นผู้เลือก "จิ้ม" ยืนยันว่ายาตัวใดคือสาเหตุของการแพ้ที่แท้จริง
Edit: สามารถแก้ไขหรือพิมพ์หมายเหตุข้อมูลเพิ่มเติมได้
Reject: หากตรวจสอบแล้วพบว่าประวัติไม่ถูกต้อง (ไม่ได้แพ้ยาจริง) สามารถกดปฏิเสธได้ โดยข้อมูลที่ถูก Reject จะต้องถูกแยกไปจัดเก็บไว้อีกตารางหนึ่ง ไม่นำมาปะปนกับข้อมูลจริง
Verify/Stamp: ข้อมูลที่ผ่านการยืนยันแล้ว จะถูกประทับตรา (Stamp) และย้ายไปเก็บใน "ตารางสุดท้าย" (Final Database) ซึ่งถือเป็นข้อมูลที่สิ้นสุดและจะไม่ถูกแก้ไข (Read-only) ยกเว้นในกรณีที่ต้องมีการพิสูจน์ทราบความจริงในภายหลังว่าไม่ได้แพ้ยา จึงจะมีสิทธิ์นำออก
3. การสร้างบัตรแพ้ยาอิเล็กทรอนิกส์ (E-Drug Allergy Card Generation)
Trigger: บัตรจะถูกสร้างขึ้นอัตโนมัติ (Generate) ก็ต่อเมื่อเภสัชกรหรือแพทย์กดปุ่มยืนยัน (Stamp) ข้อมูลแล้วเท่านั้น
Preview Feature: ก่อนกด Stamp ระบบจะต้องมีหน้าต่างแสดงผลลัพธ์ (Preview) ให้เภสัชกรเห็นหน้าตาของบัตรแพ้ยาที่จะถูกสร้างขึ้นจริง เพื่อตรวจสอบความถูกต้องก่อน
Data Fields บนบัตร: ข้อมูลบนบัตรต้องดูน่าเชื่อถือเสมือนโรงพยาบาลเป็นผู้ออกให้ โดยต้องประกอบด้วย: ชื่อแพทย์/เภสัชกรผู้ออกบัตร, ชื่อโรงพยาบาลที่ทำการรักษาอาการแพ้ยา, วันที่เข้านอนโรงพยาบาล, วันที่ได้รับยาที่แพ้, และสรุปรายละเอียดการแพ้ยา
 (อาจมีการออกบัตรหลายใบ หากพบว่าแพ้ยาหลายตัวแยกกัน
)
4. การเชื่อมต่อ API สำหรับใช้งานระดับประเทศ (National API Integration)
เป้าหมายของ API: เพื่อใช้บล็อก/ล็อก (Lock) การสั่งจ่ายยาที่ผู้ป่วยแพ้ ไปตลอดชีวิต ไม่ว่าผู้ป่วยจะไปรับบริการที่โรงพยาบาลใดในประเทศ
Infrastructure: อาจให้บริการในรูปแบบ API Service ผ่านโครงสร้างพื้นฐานเดิมที่มีอยู่แล้ว เช่น Smart ID
การเชื่อมต่อกับปลายทาง (Client Integration): ระบบนี้จะต้องเปิด API ให้กับโปรแกรมระบบสารสนเทศโรงพยาบาล (HIS / SAS) กว่า 30 Vendors ทั่วประเทศเชื่อมต่อ
Response Action: เมื่อมีการสั่งจ่ายยา ระบบ HIS ของโรงพยาบาลจะเช็คกับ API นี้ หากพบประวัติว่าตรงกับยาที่แพ้ (หรือยาในกลุ่ม Cross-reaction) API จะต้องส่งสัญญาณแจ้งเตือน และล็อกไม่ให้ทำการสั่งจ่ายยานั้นโดยเด็ดขาดแบบ 100%
5. ข้อมูลอื่นๆ สำหรับการวางแผนโครงการ (Project Scope & Scale)
ขนาดของข้อมูลตั้งต้น (Data Volume): ปัจจุบันมีข้อมูลดิบของผู้ป่วยที่เข้าข่ายแพ้ยาจากการประมวลผลย้อนหลัง 10 ปี อยู่ที่ประมาณ 20,000 - 30,000 ราย
แผนการ Roll-out: จะเริ่มนำร่องทดสอบระบบ API และหน้าจอการยืนยันในโรงพยาบาลพื้นที่ เขต 9 (เช่น บุรีรัมย์, โคราช, คูเมือง) ก่อน เมื่อระบบเสถียรจึงจะขยายผลในระดับประเทศ (กระทรวงสาธารณสุข)
ทรัพย์สินทางปัญญา: แนวคิดการสืบค้นข้อมูลการรับยาข้ามโรงพยาบาลย้อนหลัง 1 เดือนเพื่อมาจับคู่กับการเข้าแอดมิทแพ้ยา ถือเป็นนวัตกรรมใหม่ในระดับโลก ทางทีมผู้บริหารมีแผนที่จะนำ Logic นี้ไปจดสิทธิบัตร (Patent)
 Developer จึงควรเก็บรักษา Source Code และ Logic ส่วนนี้เป็นความลับตามนโยบายของโครงการ
